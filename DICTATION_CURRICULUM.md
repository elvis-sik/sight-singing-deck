# Dictation Curriculum (design of record)

A curriculum for **melodic dictation** (hear it → notate it) as a skill in its
**own right**, with its own difficulty ordering and its own melody pool — not the
sight-singing melodies run backwards.

This supersedes the "dictation reuses the sight-singing material" stance in
[`CURRICULUM.md`](CURRICULUM.md). Sight-singing stays as designed; this document
covers only the parallel dictation path.

## Why a separate curriculum

Dictation and sight-singing are the two directions of one skill (audiation), but
they are **gated by different things**, so the optimal ordering diverges:

| Axis | Sight-singing difficulty | Dictation difficulty |
|---|---|---|
| Vocal production / pitching a leap | **primary** | irrelevant (you're not singing) |
| Range / tessitura | matters | irrelevant |
| **Phrase length / working memory** | minor | **primary** — the core dictation skill |
| **Rhythmic complexity** | a separate track; melodies stay even | **primary**; isolated in its own RD track, then integrated |
| **Hearings available** | n/a (you read at your own pace) | **the hidden axis** — free replay must be countered (see below) |
| Interval **audibility** (is it easy to *hear*?) | secondary | **primary** — consonant/triadic first |
| Tonal function (which degrees) | primary early | primary early (**shared** — the one axis both agree on) |

Two consequences:

1. **Length and rhythm are first-class ramps** for dictation (they barely move in
   the early sight-singing ladder). A 6-note stepwise phrase is trivial to sing
   but taxes dictation memory; a short leapy phrase is the reverse.
2. **Reusing the same tunes degrades dictation into recall.** Once you've
   sight-sung a melody a dozen times, "taking dictation" of it is remembering, not
   hearing — and sibling-burying only prevents *same-day* collisions, not
   long-term familiarity. So dictation needs its **own melody pool**.

## Structural approach

**As built (2026-07-07)** — the goal below (a melody generates exactly one card;
dictation is separate) is realized with **two note types**, not one conditional
model:

- **Sight-singing model is Sing-only.** `make_model()` ships a single `Sing`
  template. It deliberately does *not* emit a Transcribe card: transcribing the
  very tunes you sight-sing is the "recall not hearing" trap above. (Historically
  the model shipped a second Transcribe card per melody; that was removed by
  design — build-side in `make_model`, and in the live collection by dropping the
  `Transcribe` template via `scripts/apply_live_template_update.py
  --remove-template Transcribe`.)
- **Dictation has its own note type.** `make_dictation_model()` is a single
  `Dictate` card (the transcription editor + the listen-count mechanic). One
  melody → one card, no Sing sibling — the "conditional card generation" goal
  reached with a separate model rather than role-marker fields on a shared one.
- **Two deck trees.** Sing cards under `Sight Singing::…`; Dictate cards under a
  parallel `Dictation::…` tree with its own numbered stages. Independent
  scheduling and daily volume.
- **Disjoint melody pools.** Dictation draws its own generated melodies (same
  engine, different ladder) so it stays genuine ear-work; there is no forced
  overlap with the sung tunes.

The original plan was one note type with role-marker conditional generation
(`SingEnabled` / `DictateEnabled`); the separate-note-type route was chosen
because it needs no shared-model surgery and keeps the live Sing/Transcribe deck
(already in the collection) low-risk. See [[ankiconnect-live-deck-update]].

All of this reuses the existing `Stage` constraint-spec engine
(`curriculum/stages.py`): `pool` / `start_pool` / `end_pool`, `max_step` /
`min_step`, `length`, `require_present_any`, `require_leap_min`,
`forbid_tritone_leap`, `mode`, etc. A dictation stage is just a different point in
the same parameter space, ordered differently.

## Grading & the listen-count model

**What "correct" means.** For these 1–2-bar drills the bar is a **perfect
transcription** — exact pitches (right octaves) and exact rhythm. Two grading
regimes, split by whether a stage contains rests/ties:

- **Note-only, even-rhythm stages → exact match.** With no rests and no ties, each
  event is a note of the fixed value and there is genuinely one spelling: in a
  known key each staff position is a single diatonic pitch (no enharmonic choice —
  A minor's raised 7 is just G♯), and a note of N beats has one notation (splitting
  it needs a tie, and an *untied* split is two attacks — a different sound). So
  exact match is correct and unambiguous. The entire melodic spine (DP0–DD9) is
  pitch-only and rest-free, so it stays wholly in this regime.
- **Any stage with rests or ties → compare by *sounded* rhythm.** A rest has no
  attack, so a 2-beat rest is validly written as a half rest **or** two quarter
  rests (or four eighth rests) — tie-free, identical silence — and which spelling
  is *conventional* is even position-dependent (a rest straddling the mid-bar beat
  2–3 boundary should be broken; a full bar is a whole rest). Likewise a sustained
  2-beat note may be a half note or two tied quarters. So "correct" means the same
  **sounded rhythm** (onsets + durations, rests as gaps), not the same glyph. This
  applies as soon as rests appear — **the RD track and any future pitch+rhythm
  strand**, none of which are in the pitch-only spine — not only to
  the advanced tie/triplet stages. The grader normalises both the answer and the
  learner's entry to a canonical sounded-rhythm form and compares those.

**Grading is always manual, and the buttons mean effort.** You grade every card
yourself with Anki's Again / Hard / Good / Easy, which for a skill drill correctly
encode *how much effort the recall took* — which is what the scheduler needs. So
the listen count is **not** a rival signal; it's an **advisory anchor** for that
manual grade. The card counts plays, shows the count ("Listens: 3") and a
suggested grade, and you press the button.

Free replay is otherwise the trap of self-study dictation: a 6-note phrase you can
replay forever stops being a memory test. Showing (and grading against) the count
restores the scarcity that gives the length ramp its meaning. Implementation is
small: a play counter on the Transcribe front's melody button + a suggested-grade
reminder on the back — no editor change. Applies to every dictation stage.

**Thresholds are per-stage defaults, editable.** The count→grade mapping can't be
one absolute scale (2 listens is failure on DD1, excellent on DD9), so each stage
ships **sensible defaults that scale with its difficulty**, stored on the card so a
user can edit them. Starting shape: easy early stages expect ≤1–2 listens for Good;
later stages allow more before Hard/Again.

## The melodic ladder (major)

Diatonic indices: `0`=do, `1`=re, `2`=mi, `3`=fa, `4`=so, `5`=la, `6`=ti,
`7`=do′, and `-1`=ti, (the leading tone *below* the tonic). Tonic triad =
`{0,2,4}`. Every card primes the key firmly (cadence + tonic + replayable audio);
the tritone is excluded as a bare leap, same rule as sight-singing.

**Four design rules are enforced in the spec** (not just intended — see
`tests/test_dictation_ladder.py`):

1. **One new variable per rung.** A rung raises *length* OR introduces *one*
   interval class — never both. Steps are pushed all the way to length 6 (DD5)
   *before* any leap, then length resets to 4 for thirds (DD6) and again for
   4ths/5ths (DD7). Only the capstone combines.
2. **No immediate repeats** in any pitch-only stage (`min_step ≥ 1`). In even
   rhythm a repeated pitch carries no pitch information and smuggles in an
   onset-counting (rhythm) task; whether a re-articulation is even audible depends
   on the synth envelope. Repeats return only when the rhythm strand gives them
   meaning.
3. **Interval ceiling is hard** per stage (`max_step`); `max_step ≤ 4` (a 5th)
   means the strand *never* produces a 6th.
4. **Objective coverage is asserted, not hoped for.** DD8 pairs the resolution
   rule with a required leading tone so *every* item resolves ti→do — the first
   review found the old tendency stage was titled "ti→do" yet generated none.

### Priming floor (below the old first rung)

A true beginner cannot start at 3-note sequences. Two generated 2-note rungs and a
fixed single-note rung come first:

- **DP0 · Tonic Echo** — hear the key, identify **one** degree against it (pillars
  do/mi/so). A lone note isn't a generatable "melody" (the engine rejects
  monotone), so this is a fixed 3-item set the builder emits directly.
- **DP1 · Two-Note Pillars** — two triad tones either direction (`do–mi`, `so–do`
  …): relate a second stable pitch to the tonic. The true rung 1.
- **DP2 · Two-Note Step** — one adjacent step, incl. the `mi–fa` half-step: hear
  step-size before stringing steps.

### Length ramp on steps only (isolate length from leaps)

- **DD1 · The Triad Skeleton** — the six orderings of `do–mi–so`, length 3. Leaps
  here land only on triad tones (the easy, functional case). *Triad-first is
  correct for dictation:* the pillars are the most stable perceptual anchors, and
  "steps before leaps" is a singing-production heuristic, not a hearing one.
- **DD2 · Stepwise Neighbors** — pentachord, length 3, `mi–fa` featured.
- **DD3 · The Pentachord** — pentachord, length 4, stepwise (open + closing
  shapes; a length-4 no-repeat step phrase can't close tonic→tonic — see note).
- **DD4 · Into the Upper Scale** — full scale, length 5, stepwise, reaching
  la/ti/do′ and settling.
- **DD5 · Long Stepwise Phrases** — full scale, length **6**, **steps only**.
  Length is the sole new variable before any leap. Pure chunking/memory.

### Leaps enter — length resets, one interval class at a time

- **DD6 · Skips of a Third** — **reset to length 4**; add the 3rd (the most
  audible leap), ceiling a 3rd.
- **DD7 · Fourths & Fifths** — still length 4; add P4/P5 (no tritone), landing on
  framework tones and recovering by step. Ceiling a 5th — never a 6th.

### Length grows again, then the capstone

- **DD8 · Tendency Tones (the leading tone)** — length 5. `ti→do` resolves in
  **every** item; thirds are already learned, so the leading tone may be
  approached by a ≤3rd (`fa-mi-so-ti-do′`, `so-fa-la-ti-do′`). `fa→mi` is trained
  across the stepwise stages and appears here too. No new leap class.
- **DD9 · Free Diatonic (capstone)** — length 6, steps + learned leaps (ceiling a
  5th), a tendency tone present, stepwise recovery after leaps, resolving to a
  stable tone. The exit skill → go transcribe real music.

> **Why triads before 4ths/5ths:** every card is cadence-primed, so leaps are a
> *functional* judgement — in movable-do, `do–mi–so` leaps are the easiest to
> identify by function even though bare P4/P5 are the more recognizable isolated
> intervals. The priming makes the triadic-first order correct for this deck.

> **The parity note (DD3):** with repeats forbidden, each step flips index parity,
> so a length-4 (3-move) stepwise phrase *cannot* both start and end on a triad
> tone (all even indices). DD3 therefore allows any pentachord launch/arrival —
> yielding both closing shapes (`fa-mi-re-do`) and open ones (`do-re-mi-fa`) —
> with scoring still preferring tonic arrivals. This is a real property of the
> constraint, not a fudge; the old spec only hid it by allowing repeats.

**Rhythm is a separate strand.** These are the pitch-only, even-rhythm stages;
pitch+rhythm integration and pure rhythm dictation live in the RD sub-track (see
*Parallel sub-tracks*), so nothing above trains rhythm.

**Minor (ND-series).** Mirror the above in la-based minor (natural minor
structural stages + a harmonic-minor leading-tone stage), exactly as the
sight-singing ladder mirrors major → minor. Not yet encoded for dictation.

### Wrapper additions (rev 3)

These matter as much as the ladder order and are design-of-record even where not
yet built:

- **Tonic maintenance.** From DD5 on, offer a **tonic drone** under the melody and
  **re-sound the tonic** on every replay. This is the main defense against
  students silently switching from functional hearing to intervallic chaining on
  the longer phrases.
- **Answer in solfège / scale-degree first; staff notation is a later, separate
  concern.** Notating on a staff adds clef + key-signature translation orthogonal
  to hearing; mixing them means a hearing-correct answer loses points on notation
  mechanics and the two error types can't be told apart.
- **Advancement rubric on the listen-count** (turns the rev-2 mechanic into a
  gate): advance a rung at ≤2 listens across a run; ≥5 listens ⇒ drop a rung or the
  item is mis-specced. Log the count so mis-tuned rungs are visible.
- **Interleave real folk-tune fragments from ~DD3.** Generated fragments are
  melodically sterile by construction; simple diatonic tunes are true-beginner
  dictation *and* the bridge to the stated endpoint (transcribe real music). Let
  Anki's scheduler supply retention/interleaving; the ladder supplies the order.

## Producing the exercises (generation, not hand-authoring)

Each stage's melodies are **generated** from its spec by the existing engine
(`generate/melody_gen.py`: enumerate under the pool / step / required-tone rules →
hard-rule filter → musicality score → diversity sample), exactly as the
sight-singing stages are. So "designing a stage" means writing and **tuning its
`Stage` spec**, then generating and curating the output — not authoring melodies by
hand.

The whole strand is encoded in `curriculum/stages.py` as `DICTATION_STAGES` (plus
the fixed `DICTATION_PRIMING_SINGLETONS` for DP0). Two commands close the loop:

- **`python scripts/export_dictation_specs.py`** writes `out/dictation_specs.json`
  (the specs, machine-readable / tool-consumable) and `out/dictation_exercises.json`
  (what each spec actually generates, as diatonic indices + movable-do solfège, for
  eyeball curation). Current yield: 137 exercises across the 11 generated stages.
- **`python -m unittest tests.test_dictation_ladder`** asserts the four design
  rules above hold against the *generated* output — no immediate repeats, no
  interval past a 5th, each stage's ceiling respected, and (the regression guard)
  ti→do in every DD8 item. A spec that drifts from its objective fails the build.

Per-stage tuning is still real work: specs need adjustment for *yield* (a too-tight
spec collapses — DD8 went from ~4 items to 16 once the leading-tone constraint was
separated from the over-strict "every fa must descend" default) and for *quality*
(forbidding immediate repeats via `min_step ≥ 1` also killed the old draft's dull
`do-do-re-do` noodling for free).

**Rhythm is out of scope for this spine.** The melodic generator produces even
(all-quarter) melodies only; pitch+rhythm integration and pure rhythm dictation are
a **separate strand** (the RD sub-track below) and need new generation logic. The
pitch-only spine (DP0–DD9) is complete and buildable today.

## Parallel sub-tracks

- **Interval dictation** (`IVD2…IVD8`) — hear two notes, notate the interval. **The
  first note is shown on the staff; you place only the second**, so the task is
  "hear the interval," not "guess the starting position." Dictation counterpart to
  the interval-singing drills; reuse the interval stage specs, dictation card only.
- **Rhythm dictation** (`RD1…RD9`) — the **sole home** for pure rhythm-on-one-pitch
  dictation (kept out of the melodic spine to avoid duplication): single repeated
  pitch, notate the rhythm. A full ladder paralleling the sight-singing Rhythm
  track. Start it **in parallel** with the melodic spine (around DD5, once phrases
  lengthen) so rhythm is trained on its own before any future pitch+rhythm strand
  asks you to integrate the two. Basic values use the editor's existing duration
  toolbar + rest mode; the advanced values need editor work (see below).
- **Transfer** (other keys, bass clef) — later, same as sight-singing; dictation
  in a new key/clef is a distinct and worthwhile skill.

## Editor capability (verified 2026-07-07)

The Transcribe editor **already accepts duration input**: it has a duration
toolbar (`ss-durbar`) with **whole / half / quarter / eighth** values and a
**rest** mode (`assets/_transcription.js`: `DURATION_ORDER`, `DURATION_LABELS`,
`mode: note|rest|erase`). So basic rhythm dictation — the basic **RD** stages and
any future pitch+rhythm strand — needs no editor changes for *input*. (Grading is a
separate matter: any stage with rests needs the sounded-rhythm equivalence grader
— see *Grading & the listen-count model*.)

What the editor does **not** yet input is the *advanced* rhythmic values:
**dotted notes, ties/syncopation, and triplets** (`DURATION_ORDER` is only
`["w","h","q","8"]`). Those are needed only for the advanced Rhythm-Dictation
stages (the RD equivalents of the sight-singing R6–R9). Add them to the editor
before those specific stages; everything else is unblocked.

### Multi-bar editor (Phase 1a — DONE 2026-07-07)

The transcription editor was originally single-bar: `targetEventsFromData`
returned `[]` unless the events summed to exactly `BAR_UNITS` (8 eighth-units = 4
beats), so any melody that wasn't 4 quarters showed *"outside the current
transcription exercise scope."* That rejected the whole variable length ramp
(DP0–DP2, DD1/DD2, DD4/DD5/DD8/DD9) — **and** it silently broke the live
sight-singing Transcribe cards for the length-5/6 stages (M2/M6/M8/M9 + minor),
which were shipping un-answerable.

**Fix:** generalized the editor's single hardcoded `BAR_UNITS` into a dynamic
`capacity` = the target melody's full length (beat-aligned), set from the target
in `initFront`/`renderSummary`/`SightSingingTranscriptionReview`. The whole
melody lays out as one extended staff; a 4-quarter melody keeps `capacity === 8`
so single-bar behaviour is byte-for-byte unchanged. The occupied-grid, placement
tiles, beat bar (now N cells), status copy, and the empty-target fallback all key
off `capacity`. `targetEventsFromData` now accepts any well-formed melody. The
one subtlety: the review path must set `capacity` **before** `loadSavedEvents`,
or a multi-bar answer's bar-2 events (unit ≥ 8) get filtered out.

**Verified in-browser** (Chromium): DD5 (len-6) renders a 6-beat staff and
accepts notes in bar 2 (units 8, 10); M9 (len-6 sight-singing) front→back grades
"Perfect"; single-bar M1 still shows 4 beats and grades a wrong note as partial.
Automated: a new Playwright test drives a 6-quarter harness (6 beat cells, notes
past bar 1, "All 6 beats filled"); the single-bar suite (Chromium + WebKit) stays
green. The full ladder is now answerable end to end. *(Cosmetic follow-up: the
extended staff draws no interior barlines — fine for even-rhythm pitch entry.)*

## Phasing

1. **Phase 1 — melodic spine, major** (DP0–DD9, pitch-only) **plus the listen-count
   mechanic**. **Done.** Stages encoded + test-guarded; the dictation note type
   (`make_dictation_model`, single Dictate card + listen-count), the build
   (`scripts/build_dictation_deck.py` → `out/dictation-curriculum.apkg`, own
   `Dictation` deck tree), the listen-count UI, and **Phase 1a — multi-bar editor
   input** (above) are all done and verified end to end. The whole ladder is
   answerable; not yet imported into a real collection (a Docker/Xvfb deck smoke +
   a study pilot are the remaining validation).
2. **Phase 2 — minor** (ND-series).
3. **Phase 3 — interval dictation + the basic Rhythm-Dictation ladder** (RD, basic
   values). RD is independent and can be practised earlier, but this is when it's
   built out.
4. **Phase 4 — advanced rhythm dictation** (dotted / ties / triplets): after the
   editor's duration palette gains those values.
5. **Phase 5 — transfer** (other keys, bass clef).

## Decisions & remaining scope

Resolved (2026-07-07):

- **Card-off mechanism** — conditional card generation (a melody in one curriculum
  generates only that card); not suspension.
- **Overlap policy** — near-zero; overlap only where a tune is deliberately wanted
  both ways.
- **Tree layout** — `Dictation` is its **own top-level tree**, parallel to
  `Sight Singing`, with its own numbered tracks.
- **Grading** — perfect-transcription, exact match (sounded-rhythm equivalence only
  in the advanced-rhythm stages); manual grade; listen count as an advisory anchor;
  **per-stage, user-editable thresholds**.
- **Target learner** — blank beginner (the deck ships publicly).
- **Difficulty ramp is the author's job**, not the pilot's — a deliberately,
  finely-graded ladder (a priming floor, then length stepped 2→3→4→5→6 on steps
  only, resetting to 4 when a leap class enters so only one axis moves per rung).
  The pilot fine-tunes; it does not design.
- **No adaptivity.** A performance-driven add-on (dynamically creating/suspending
  cards, as commercial audiation trainers do) would be more powerful but is out of
  scope. A fixed, well-graded ladder builds the *basic* skills time-effectively;
  the learner then graduates to real music. Implication: the curriculum has a clear
  **endpoint** ("you can take dictation of a simple diatonic tune at tempo — go
  transcribe real music"), it does not sprawl.

Remaining (I choose sensible defaults; pilot tunes):

- Exact per-stage **threshold values** and **length steps**.
- **Sounded-rhythm equivalence grader** — needed as soon as **rests** appear (the
  RD track and any future pitch+rhythm strand), because a rest decomposes into
  smaller rests tie-free with identical sound. Not an advanced-only concern; it's a
  prerequisite for the first rhythm-bearing stages. (The pitch-only spine DP0–DD9
  doesn't need it.)
- **Advanced-rhythm editor input** (dotted / tie / triplet values in the duration
  palette) — a separate, later item for the advanced RD stages only.
